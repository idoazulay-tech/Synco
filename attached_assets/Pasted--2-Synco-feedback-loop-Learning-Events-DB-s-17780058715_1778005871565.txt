בדוק במיוחד:

2. האם היא שומרת ל-DB או רק InMemory.

החלטה ארכיטקטונית:
המטרה היא DB-backed learning events.
לא InMemory.

אם קיימת route /api/learning/events והיא InMemory:
אל תמחק אותה.
עדכן אותה או צור route חדש בצורה בטוחה כך שהיא תשמור ל-PostgreSQL.

מודל Prisma מומלץ, רק אם אין קיים מודל מתאים:

model LearningEvent {
  id              String   @id @default(cuid())
  userId          String
  taskId          String?
  eventType       String
  source          String?
  occurredAt      DateTime @default(now())
  dateIso         String?
  taskTitleSnapshot String?
  fromStatus      String?
  toStatus        String?
  fromStartTime   DateTime?
  toStartTime     DateTime?
  fromEndTime     DateTime?
  toEndTime       DateTime?
  metadata        Json?
  createdAt       DateTime @default(now())

  @@index([userId])
  @@index([taskId])
  @@index([eventType])
  @@index([occurredAt])
}

אם PostgreSQL/Prisma בפרויקט לא מקבל Json בצורה הזאת, התאם לפי המבנה הקיים.
אל תשבור schema קיים.

Backend:
צור או עדכן endpoint:

POST /api/learning/events

body:
{
  "userId": "default-user",
  "taskId": "task-id",
  "eventType": "task_completed",
  "source": "taskStore",
  "dateIso": "2026-05-05",
  "taskTitleSnapshot": "שם המשימה",
  "fromStatus": "pending",
  "toStatus": "completed",
  "fromStartTime": "2026-05-05T07:00:00.000Z",
  "toStartTime": "2026-05-05T07:00:00.000Z",
  "metadata": {
    "duration": 30,
    "priority": "high"
  }
}

Validation:
1. userId חובה.
2. eventType חובה.
3. eventType חייב להיות אחד מ:
   - task_created
   - task_completed
   - task_rescheduled
   - task_deleted
   - schedule_applied
   - task_updated
4. taskId אופציונלי, כי schedule_applied יכול לכלול כמה משימות.
5. metadata אופציונלי.
6. אם חסר userId או eventType, החזר 400.
7. אם יש שגיאת DB, החזר 500 עם הודעה פשוטה.

Response מוצלח:
{
  "ok": true,
  "event": {
    "id": "...",
    "eventType": "task_completed",
    "taskId": "...",
    "occurredAt": "..."
  }
}

הוסף גם:

GET /api/learning/events?userId=default-user&limit=50

שיחזיר את האירועים האחרונים.
זה לצורך בדיקה בלבד, לא UI רחב.

הלוגינג חייב להיות safe:
אם שמירת learning event נכשלת, אסור לשבור את פעולת המשתמש.
לדוגמה:
אם המשתמש סימן משימה כבוצעה וה-event נכשל, המשימה עדיין צריכה להישאר completed.
רק להציג console.warn או טיפול שקט.

חיבור לאירועים:

1. task_created
כאשר משימה חדשה נשמרת בהצלחה ב-DB:
שמור event:
eventType: task_created
source: taskStore או AddTaskPage
taskId
taskTitleSnapshot
toStatus

2. task_completed
כאשר משתמש מסמן משימה כבוצעה:
שמור event:
eventType: task_completed
fromStatus: pending
toStatus: completed
taskId
taskTitleSnapshot
metadata:
  duration
  priority
  flexibility

3. task_deleted
כאשר משתמש מוחק משימה:
שמור event:
eventType: task_deleted
fromStatus
toStatus: deleted

4. task_rescheduled
כאשר משתמש משנה זמן של משימה ידנית, כולל Drag & Drop אם יש action קיים:
שמור event:
eventType: task_rescheduled
fromStartTime
toStartTime
fromEndTime
toEndTime
source: drag_and_drop או edit_task

אם קשה לחבר את Drag & Drop בלי סיכון:
אל תשבור אותו.
חבר רק למקום המרכזי שבו taskStore מעדכן startTime/endTime.
דווח אם Drag & Drop משתמש בנתיב אחר.

5. schedule_applied
כאשר המשתמש לוחץ "אשר סידור ושמור" וה-apply-schedule הצליח:
שמור event אחד:
eventType: schedule_applied
source: PlanMyDayButton
metadata:
  updatedCount
  taskIds
  date
  dayLoadMinutes אם קיים
  priorityScores אם קיימים

חשוב:
לא לשמור schedule_applied לפני שה-DB אישר הצלחה.

אם עדיף לוגינג של schedule_applied בצד backend בתוך /api/planner/apply-schedule:
עשה זאת שם, כי זה אמין יותר.
אבל אל תיצור כפילות גם backend וגם frontend.
בחר מקום אחד.

המלצה:
- task_created/task_completed/task_deleted/task_rescheduled אפשר דרך user-tasks route או taskStore, לפי מה שהכי בטוח.
- schedule_applied עדיף בתוך backend apply-schedule אחרי transaction מוצלח.

## 2. החלטה ארכיטקטונית

## 3. מה שונה

## 4. מה גובה

קובץ מקורי | קובץ גיבוי

## 5. אילו אירועים נשמרים עכשיו

פעולת משתמש

→ GET /api/learning/events מציג היסטוריה
