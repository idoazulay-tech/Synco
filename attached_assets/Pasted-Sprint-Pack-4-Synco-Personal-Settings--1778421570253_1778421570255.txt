Sprint Pack 4א בפרויקט Synco:
Personal Settings בסיסי לתכנון מותאם משתמש.

מטרה:
להוסיף שכבת הגדרות משתמש בסיסית שמשפיעה על התכנון של סינקו בצורה בטוחה:
- שם משתמש מקומי
- timezone
- dayStart
- dayEnd
- planning buffer
- transition buffer
- planning style
- dark mode preference אם כבר קיים מנגנון מתאים
- reset learning data רק באזור devtools או settings מתקדם

המטרה אינה לבנות מערכת משתמשים מלאה.
אין Auth בשלב הזה.
הכול עדיין תחת default-user, אבל עם UserSettings אמיתי וברור.

חשוב מאוד:
אל תוסיף Auth.
אל תוסיף Google Calendar.
אל תוסיף Qdrant.
אל תוסיף AI.
אל תבנה dashboard חדש.
אל תשנה DB בצורה מסוכנת.
אל תשנה schedulePlanner עמוק.
אל תשנה buildSchedule.
אל תשנה priorityScoreEngine אלא אם צריך רק לקרוא settings.
אל תשבור planner/schedule.
אל תשבור planner/apply-schedule.
אל תשבור planner/commit-draft-schedule.
אל תשבור PlanningDraftPanel.
אל תשבור PlanMyDayButton.
אל תשבור DayView.
אל תשבור Drag & Drop.

אל תשבור Trust QA script.
כל שינוי חייב להיות קטן, מבודד, ובדוק.

חלק 1: מודל Settings

אם כבר קיים UserSettings ב-Prisma:
השתמש בו.
אל תיצור מודל כפול.

אם חסרים שדות, הוסף רק אם צריך ובזהירות:

שדות מומלצים:
- userId String unique/index
- displayName String?
- timezone String? default "Asia/Jerusalem"
- dayStart String? default "08:00"
- dayEnd String? default "22:00"
- planningBufferMinutes Int default 5
- transitionBufferMinutes Int default 5
- planningStyle String default "balanced"
- themePreference String? default "system"
- createdAt DateTime
- updatedAt DateTime

planningStyle ערכים:
- gentle
- balanced
- aggressive

הסבר:
gentle = תכנון עם יותר מרווחים ופחות עומס.
balanced = ברירת מחדל מאוזנת.
aggressive = ניצול יותר חזק של חלונות זמן.

אם schema כבר שונה בעבר ויש UserSettings עם Json, אפשר להשתמש ב-Json במקום migration רחב.
בחר את הדרך הכי פחות מסוכנת.

חלק 2: API Settings

צור או עדכן routes:

GET /api/settings?userId=default-user

אם אין settings:
החזר defaults:
{
  ok: true,
  settings: {
    userId: "default-user",
    displayName: "",
    timezone: "Asia/Jerusalem",
    dayStart: "08:00",
    dayEnd: "22:00",
    planningBufferMinutes: 5,
    transitionBufferMinutes: 5,
    planningStyle: "balanced",
    themePreference: "system"
  }
}

PUT /api/settings

body:
{
  userId,
  displayName,
  timezone,
  dayStart,
  dayEnd,
  planningBufferMinutes,
  transitionBufferMinutes,
  planningStyle,
  themePreference
}

Validation:
1. userId חובה.
2. dayStart/dayEnd בפורמט HH:MM.
3. dayStart חייב להיות לפני dayEnd.
4. planningBufferMinutes בין 0 ל-60.
5. transitionBufferMinutes בין 0 ל-60.
6. planningStyle רק gentle/balanced/aggressive.
7. themePreference רק system/light/dark.
8. timezone string לא ריק.

Response:
{
  ok: true,
  settings
}

חלק 5: שילוב ב-planner/schedule

ב-server/routes/planner.ts:

כאשר מתקבלת בקשה ל-/api/planner/schedule:
1. טען settings לפי userId.
2. אם אין settings, השתמש defaults.
3. בנה config:
   - dayStart מתוך settings.dayStart
   - dayEnd מתוך settings.dayEnd
   - buffer מתוך settings.planningBufferMinutes אם רלוונטי
   - transitionBuffer מתוך settings.transitionBufferMinutes אם רלוונטי
   - planningStyle משפיע רק באופן עדין

חשוב:
No Past Scheduling עדיין מנצח.
אם היום הנוכחי והשעה עכשיו אחרי dayStart:
earliestAllowedTime = now + planningBufferMinutes
ולא dayStart.

כלומר:
settings.dayStart = 08:00
now = 17:20
planningBuffer = 5
earliest = 17:25

חלק 6: planningStyle

יישום ראשוני, בלי שינוי עמוק:

gentle:
- הוסף warning/summary:
  "מצב תכנון עדין פעיל, סינקו ישאיר יותר מרווחים"
- אם יש config מתאים ל-buffer, הגדל planningBufferMinutes ב-5.
- אל תשנה schedulePlanner עמוק.

balanced:
- השתמש בערכים כמו שהם.

aggressive:
- הקטן buffer בפועל ב-2 דקות, אבל לא מתחת ל-0.
- אל תדחוס משימות מעבר למה שה-planner כבר עושה.
- אל תשבור No Past.

אם אין דרך בטוחה להשפיע על השיבוץ כרגע:
רק החזר planningStyleSummary ב-response ודווח.
