חלק 2: Duration Intelligence Service

צור שירות חדש:

server/services/durationIntelligenceService.ts

השירות ייצא פונקציה:

buildDurationSuggestionContext(userId: string, tasks: any[], options?: {
  lookbackDays?: number,
  dateIso?: string
})

הפונקציה תחזיר:

{
  enabled: true,
  lookbackDays: 14,
  suggestions: [
    {
      taskId: "task-id-or-draft-id",
      title: "שם המשימה",
      currentDurationMinutes: 20,
      suggestedDurationMinutes: 45,
      confidence: "low" | "medium" | "high",
      sampleSize: 3,
      averageActualDurationMinutes: 45,
      averageDeltaMinutes: 25,
      reason: "משימות דומות לקחו לך בפועל יותר זמן מהמתוכנן"
    }
  ],
  warnings: []
}

כללים:
1. lookbackDays ברירת מחדל: 14.
2. קוראים LearningEvents מסוג task_execution_completed.
3. משתמשים רק באירועים שיש להם metadata.actualDurationMinutes.
4. מתעלמים מ-source:
   - test
   - seed
   - debug
   - manual_test

5. התאמת משימות:
   שלב ראשון פשוט ובטוח:
   - אם taskId זהה למשימה קיימת, התאמה חזקה.
   - אם אין taskId זהה, אפשר התאמת כותרת בסיסית רק אם יש דמיון גבוה מאוד.
   - כדי למנוע זיכרון שקרי, title matching חייב להיות conservative.
   - אם אין ודאות, לא מציעים suggestion.

6. התאמת כותרת בטוחה:
   אפשר להשתמש בפונקציה פשוטה:
   normalizeTitle(title)
   remove common words בעברית כמו:
   את, של, עם, על, לפני, אחרי, ל, ב, ה
   ואז להשוות מילים משמעותיות.
   חובה לפחות 2 מילים משמעותיות משותפות או התאמה כמעט מלאה.
   אם אין מספיק מילים משותפות, אין suggestion.

7. confidence:
   - sampleSize >= 5 → high
   - sampleSize 3-4 → medium
   - sampleSize 1-2 → low

8. לא להציע שינוי אם:
   - currentDurationMinutes חסר
   - suggestedDurationMinutes קטן מ-1
   - הפער קטן מ-5 דקות
   - sampleSize = 0

9. suggestedDurationMinutes:
   עיגול ל-5 דקות הקרובות.
   לדוגמה:
   43 → 45
   47 → 45 או 50 לפי פונקציה עקבית.

10. לא כותבים שום דבר ל-DB בשירות הזה.
הוא read-only בלבד.

חלק 3: endpoint בדיקה

בתוך server/routes/learning.ts או route מתאים אחר, הוסף endpoint:

GET /api/learning/duration-suggestions?userId=default-user&date=2026-05-09

הוא יחזיר context כללי לפי events אחרונים.

אם אין tasks בקלט, אפשר להחזיר:
{
  ok: true,
  context: {
    enabled: true,
    lookbackDays: 14,
    suggestions: [],
    warnings: []
  }
}

אם נוח יותר לקבל tasks ב-POST, הוסף:

POST /api/learning/duration-suggestions

body:
{
  userId,
  dateIso,
  tasks
}

עדיף POST כי צריך לבדוק מול משימות קיימות או טיוטות.

חלק 4: שילוב ב-planner/schedule

ב-POST /api/planner/schedule:

1. אחרי rawTasks ולפני response:
   בנה durationSuggestionContext:
   buildDurationSuggestionContext(userId, rawTasks, { dateIso })

2. אל תשנה duration בפועל.
3. אל תשנה את השיבוץ אוטומטית לפי ה-suggestion בשלב הזה.
4. רק הוסף ל-response:
   durationSuggestionSummary:
   {
     enabled: true,
     lookbackDays: 14,
     suggestionCount: N,
     warnings: [...]
   }

5. בכל scheduledTask הוסף אם קיים:
   durationSuggestion:
   {
     currentDurationMinutes,
     suggestedDurationMinutes,
     confidence,
     sampleSize,
     averageActualDurationMinutes,
     averageDeltaMinutes,
     reason
   }

חשוב:
אם duration service נכשל, schedule עדיין חייב לעבוד.
הוסף warning:
"לא ניתן היה לחשב הצעות משך כרגע"

חלק 7: Execution Start Expiry Guard קטן

ב-src/store/taskStore.ts:

הוסף ניקוי עדין ל-executionStartTimes ישנים.

כלל:
אם executionStartTime ישן יותר מ-24 שעות:
- אל תשתמש בו לחישוב actualDurationMinutes.
- מחק אותו מה-store.
- כאשר completeTask קורה אחרי expiration, fallback להתנהגות הקיימת.
- הוסף metadata:
  actualStartSource: "expired_execution_start_fallback"

מטרה:
למנוע מצב שמשתמש התחיל משימה לפני שבוע ולא סיים, ואז duration יוצא לא הגיוני.

אל תבנה UI חדש לזה.
