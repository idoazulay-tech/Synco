חלק 9: אישור Preview של DraftTasks

אסור להפעיל apply-schedule על ids זמניים של DraftTasks.

צריך ליצור UserTasks אמיתיות ורק אז לשמור את השיבוץ.

הפתרון המועדף:
צור endpoint חדש אטומי:

POST /api/planner/commit-draft-schedule

body:
{
  "userId": "default-user",
  "date": "2026-05-08",
  "draftTasks": [...],
  "scheduledItems": [
    {
      "draftId": "...",
      "startTime": "17:35",
      "endTime": "17:45"
    }
  ],
  "timezoneOffsetMinutes": -180,
  "userTimeZone": "Asia/Jerusalem"
}

ה-endpoint יעשה transaction:
1. validate userId/date/draftTasks/scheduledItems
2. create UserTasks אמיתיות מה-draftTasks
3. להתאים draftId ל-userTaskId אמיתי
4. לשמור startTime/endTime לפי scheduledItems
5. ליצור task_created learning events כמו שהמערכת עושה היום, או להשתמש באותו service כדי לא ליצור כפילות
6. להחזיר:
{
  ok: true,
  createdCount,
  createdTasks,
  draftToTaskMap
}

חשוב:
אם יש שגיאה באמצע:
- rollback מלא
- לא להשאיר חצי משימות ב-DB

אם קשה ליצור endpoint כזה בשלב הזה:
עצור ודווח.
אל תבצע partial save דרך כמה קריאות בלי transaction.

חלק 10: DB drafts אופציונלי

אם בוחרים DB-backed drafts, צור Prisma model:

model PlanningDraft {
  id          String   @id @default(cuid())
  userId      String
  sessionId   String?
  title       String
  description String?
  payload     Json
  status      String   @default("draft")
  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt

  @@index([userId])
  @@index([sessionId])
  @@index([status])
}

Routes:
POST /api/planning-drafts
GET /api/planning-drafts?userId=default-user
DELETE /api/planning-drafts/:id

אם זה גדול מדי:
השתמש ב-localStorage זמני ודווח.
